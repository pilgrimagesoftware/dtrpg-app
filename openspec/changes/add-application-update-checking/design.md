## Context

`dtrpg-app/rust` publishes packaged binaries via two CI paths that both call the reusable `.github/workflows/package.yaml`:
`nightly.yaml` (tag `nightly`, prerelease) and `release.yaml` (computed semver tag, non-prerelease). Both end in an
`ncipollo/release-action` step that uploads `dist/*` as release assets. Neither publishes anything describing "what is
the latest version" in a form the desktop app can cheaply fetch and compare against.

The GPUI shell already has:
- A native app menu built in `build_menus` (`crates/dtrpg-ui/src/ui/app/mod.rs`) with `About`/`Settings`-style
  `MenuItem::action` entries dispatched through the shared command model.
- A notification banner (`notification_banner_view.rs`) driven by `AuthStateController` (`controllers/auth_state.rs`),
  which owns a `Vec<Notice>` of `NoticeKind::{NotSignedIn, SessionExpired}` with a single `NoticeAction::OpenSettings`.
  The banner rendering (`render_notification_banner`) is generic over notices; only the controller and data model are
  auth-specific today.
- `reqwest` (blocking feature, rustls-tls) already a workspace dependency used by `dtrpg-core`, so no new HTTP client
  crate is needed for the update check itself.

## Goals / Non-Goals

**Goals:**
- Let a user manually trigger an update check from the app menu.
- Automatically check for updates on startup and periodically, without blocking UI or startup time.
- Show a dismissible banner using the existing notification banner mechanism when an update is available.
- Publish a small, stable, versioned manifest as part of the existing nightly/release CI so the app has one fetch
  target instead of parsing GitHub's release-list API response shape.

**Non-Goals:**
- No in-app auto-download or self-install of updates (no Sparkle/Squirrel-style updater in this change); the banner
  links out to the release page/asset.
- No update channel selection UI (stable vs. nightly) beyond following whatever channel the running build already is.
- No change to the packaging formats or platforms already established in `add-release-packaging-workflow`.

## Decisions

- **Manifest format and location**: Publish `manifest.json` as an additional `dist/` artifact uploaded alongside the
  platform packages in `package.yaml`'s existing `ncipollo/release-action` step, so it lands at a predictable URL:
  `https://github.com/pilgrimagesoftware/dtrpg-app/releases/download/<tag>/manifest.json`. Contents: `version`, `tag`,
  `published_at`, and an `assets` map of platform key to download URL. Generated by a small step in the `package` job
  (not a separate crate) using the same version/tag inputs already passed into the workflow, after the matrix jobs
  produce their `dist/` outputs.
  - Alternative considered: query the GitHub Releases REST API directly from the app at runtime. Rejected because it
    requires parsing asset names to infer platform/format, is a versioned external contract outside our control, and
    hits stricter unauthenticated rate limits than a single static asset fetch.

- **Version comparison**: Compare using `semver`-style parsing of the running build's `CARGO_PKG_VERSION` against the
  manifest's `version`. Nightly builds (detected via a compile-time flag set only by `nightly.yaml`'s build) compare
  against the `nightly` tag's manifest by `published_at` instead of semver, since nightly reuses the same version
  number across pushes. Add the `semver` crate as a new dependency for tagged-release comparison.

- **Extend the existing notice model rather than build a parallel one**: Add `NoticeKind::UpdateAvailable` and
  `NoticeAction::OpenUpdateUrl(String)` to `data/notification.rs`, and introduce a small `UpdateCheckController`
  (mirroring `AuthStateController`'s shape) that owns update-check state and contributes to the same
  `active_notices()`-style aggregation surfaced to `render_notification_banner`. This keeps one banner rendering path
  and one dismiss mechanism instead of two competing notice systems.
  - Alternative considered: a separate always-on-top toast/dialog for updates. Rejected — the notification banner
    already exists, is visually established, and dismiss/action semantics are already solved there.

- **Fetch scheduling**: Startup check fires once after the shell finishes initial session restoration (piggybacking on
  the same lifecycle point auth-state checks already use), plus a periodic recheck on a fixed interval (e.g. every 24h
  of continuous runtime) via a GPUI background task. The manual "Check for Updates" menu action always performs an
  immediate fetch regardless of the periodic timer, and is the only path that surfaces "you're up to date" or a fetch
  error — the automatic path stays silent (log only) on failure or when already up to date.

- **New menu item placement**: Add `MenuItem::action(t!("menu.app_check_for_updates"), CheckForUpdates)` to the app
  menu (first `Menu::new` block in `build_menus`) directly below `About`, matching the conventional macOS placement.

## Risks / Trade-offs

- [Risk] GitHub release assets can take a few seconds to become available after a release is created; a check
  immediately following a fresh tag could race the manifest upload. → Mitigation: the manifest is uploaded in the same
  `ncipollo/release-action` step as the platform packages, so it's atomic with the release becoming visible; no
  separate follow-up step to race.
- [Risk] Unauthenticated requests to `github.com` release asset URLs are still subject to GitHub's general rate
  limiting under heavy automated polling. → Mitigation: 24h periodic interval plus manual-only immediate checks keeps
  per-install request volume low; no per-minute polling.
- [Risk] Extending the shared `Notice`/`NoticeAction` enums couples the auth and update-check controllers through one
  shared data type. → Mitigation: both variants are simple, stateless payloads; if a third notice source appears and
  the enum grows unwieldy, revisit by giving each controller its own notice type behind a shared trait at that point
  (not preemptively).
- [Risk] The Windows packaging leg in `package.yaml` runs with `continue-on-error: true` (unverified against gpui per
  the release-packaging change), so a manifest step depending on all matrix legs succeeding must not block release
  publishing when only Windows fails. → Mitigation: manifest generation reads whichever `dist/` outputs actually
  succeeded and omits missing platform entries from the `assets` map, rather than failing the job.

## Migration Plan

- Ship CI manifest publishing first (additive: new file uploaded, no existing workflow behavior changed) and verify
  a manifest lands correctly on the next nightly run before wiring the app to fetch it.
- Ship the app-side check behind the same release cadence as any other feature; no data migration or rollback beyond
  reverting the app change, since the manifest asset is purely additive to existing releases.

## Open Questions

- Should the periodic re-check interval be configurable via settings, or fixed at 24h for this change? Defaulting to
  fixed 24h; revisit only if users ask for control.
